# 要件定義書の作成思考と学習ポイント

## 作成時の思考プロセス

### 1. 学習目的の明確化

**思考**: 「HTML/CSS基礎」という学習内容から、**初学者が基本を体系的に学べる**プロジェクトである必要がある。

#### なぜポートフォリオサイトなのか？
- ToDoリストのような単一機能アプリではなく、**複数のセクションを持つサイト**にすることで、HTML構造の理解を深められる
- ポートフォリオサイトは**実用的**で、学習後も自分の資産として使える
- 多様なコンテンツタイプ（テキスト、画像、リスト）を扱える

**学習者への教訓**: プロジェクト選定は「何を学ぶか」から逆算する。技術習得と実用性の両立が理想。

### 2. 段階的な学習設計

**思考**: dev-schedule.csvを見ると、第1週は「HTML/CSS基礎」、第2週が「レスポンシブデザイン」と続く。つまり、**今週はレスポンシブを含めない**設計にすべき。

#### スコープの制限
制約事項に明記：
```markdown
- JavaScriptは使用しない（HTML/CSSのみ）
- レスポンシブデザインは次週以降で対応
- バックエンド機能は含まない
```

**理由**:
- すべてを一度に学ぼうとすると混乱する
- 基礎を固めてから応用に進む
- **制約を設けることで、焦点を絞って深く学べる**

**学習者への教訓**: 「できること」をすべて詰め込まない。今週の学習目標に集中する。完璧主義の罠を避ける。

### 3. 具体的な技術要素の選定

**思考**: 「HTML/CSS基礎」で学ぶべき核心的な概念は何か？

#### HTMLについて
- **セマンティックタグ**: `<header>`, `<nav>`, `<section>`, `<main>`, `<footer>`
  - 意図: 単なる`<div>`の羅列ではなく、意味のある構造を学ぶ
  - ページ構造の論理的な理解
  - SEO、アクセシビリティに影響する重要な概念

**悪い例**:
```html
<div class="header">
  <div class="nav">...</div>
</div>
```

**良い例**:
```html
<header>
  <nav>...</nav>
</header>
```

#### CSSについて
- **Flexbox**: 最も基本的で実用的なレイアウト手法
  - GridよりもFlexboxを先に学ぶべき（段階的な学習）
  - ナビゲーション、スキル表示、プロジェクトカードなど、複数箇所で練習できる
  - 1次元レイアウト（行または列）に最適

**学習者への教訓**: 技術選択には理由がある。「新しい技術」より「基本的な技術」を優先する。

### 4. 適切なスコープ設定

**思考**: 初学者が**1週間で完成できる**ボリュームにする必要がある。

#### 避けた要件（大きすぎる）
- ❌ 複数ページ構成
- ❌ 複雑なアニメーション
- ❌ バックエンド連携
- ❌ ユーザー認証
- ❌ データベース連携

#### 採用した要件（適切）
- ✅ シングルページ（セクション分け）
- ✅ 基本的なホバーエフェクト
- ✅ 静的コンテンツのみ
- ✅ ページ内リンク（スムーススクロール）

**スコープ設定の基準**:
1. 学習目標を達成できるか？
2. 1週間で完成できるか？
3. 実用的な成果物になるか？
4. 次週以降に拡張できるか？

**学習者への教訓**: スコープ管理はプロジェクト成功の鍵。「できること」と「やるべきこと」は違う。

### 5. 実用性と汎用性

**思考**: 「作って終わり」ではなく、**実際に使える成果物**にする。

#### ポートフォリオサイトの実用性
- 就職活動や営業活動で実際に使える
- 自己紹介、スキル、プロジェクト紹介という構成は汎用的
- 後から内容を自分用にカスタマイズできる
- GitHub Pagesなどで公開可能

#### 学習の継続性
- 次週でレスポンシブ対応を追加できる
- その次の週でJavaScriptの機能を追加できる
- 成長に合わせて進化させられる

**学習者への教訓**: 学習プロジェクトも「使い捨て」にしない。実用的な成果物は学習のモチベーションを高める。

### 6. 学習の可視化

**思考**: 何を学んだかを明確にするため、「学習目標」セクションを設けた。

```markdown
## 6. 学習目標

このプロジェクトを通じて以下を習得します：

- HTML5のセマンティックタグの理解と使用
- CSSセレクタの基本
- Flexboxレイアウトの基礎
- カラーとタイポグラフィの基本
- ボックスモデルの理解
- 基本的なCSSアニメーション
```

**効果**:
- 学習者は「何ができるようになるか」を意識しながら進められる
- 完成時に「何を習得したか」を確認できる
- 面接やポートフォリオで「このプロジェクトで学んだこと」を説明しやすい

**学習者への教訓**: 学習目標を明文化することで、漠然とした学習を構造化できる。

## 学習者がくみ取るべき内容

### 1. 要件定義の構造理解

#### 必須セクション
1. **プロジェクト概要** - 何を作るのか、なぜ作るのか
2. **機能要件** - 具体的に何ができる必要があるのか
3. **技術要件** - どの技術を使うのか
4. **非機能要件** - パフォーマンス、デザイン、アクセシビリティ
5. **制約事項** - 何をやらないのか（スコープの明確化）
6. **成果物** - 何を納品するのか
7. **学習目標** - 何を習得するのか

**重要な原則**: 要件定義は「やること」だけでなく、**「やらないこと」も明確にする**。

#### なぜ「制約事項」が重要なのか？

**制約がない場合**:
```
「ポートフォリオサイトを作る」
↓
JavaScriptも入れよう
↓
レスポンシブも対応しよう
↓
バックエンドも...
↓
完成しない、学習が浅くなる
```

**制約がある場合**:
```
「HTML/CSSのみでポートフォリオサイトを作る」
↓
HTMLの構造を深く理解
↓
CSSのレイアウトを深く理解
↓
基礎を確実に習得
```

**学び**: スコープを限定することで、深い学習が可能になる。

### 2. セマンティックHTMLの重要性

#### なぜ`<div>`ではなく`<header>`を使うのか？

**技術的な理由**:
- **SEO**: 検索エンジンがページ構造を理解しやすい
- **アクセシビリティ**: スクリーンリーダーが正しく読み上げられる
- **保守性**: コードを見た人が構造を理解しやすい

**実例**:
```html
<!-- 悪い例: 意味がない -->
<div class="header">
  <div class="navigation">
    <div class="menu-item">About</div>
  </div>
</div>

<!-- 良い例: 意味がある -->
<header>
  <nav>
    <a href="#about">About</a>
  </nav>
</header>
```

**学び**: HTMLタグは見た目ではなく、**意味（セマンティクス）**で選ぶ。

### 3. 段階的な学習の重要性

#### 学習の順序が重要な理由

**間違った順序**:
1. HTML + CSS + JavaScript + レスポンシブ + バックエンドを同時に学ぶ
   → すべてが中途半端になる

**正しい順序**:
1. HTML/CSSの基礎 → 確実に習得
2. レスポンシブデザイン → モバイル対応を学ぶ
3. JavaScript基礎 → インタラクションを追加
4. バックエンド → データ連携

**学び**:
- 基礎を固めてから応用に進む
- 一度に多くを学ぼうとしない
- **制約を設けることで、焦点を絞って深く学べる**

### 4. ユーザー視点の設計

#### なぜ「対象ユーザー」を明記するのか？

```markdown
### 対象ユーザー
- 採用担当者
- クライアント
- 同業者・ネットワーキング相手
```

**効果**:
- 採用担当者が見やすい構成を意識する
- 専門用語を使いすぎない（一般の人にも伝わる）
- 視覚的に魅力的なデザインを心がける

**学び**: 技術学習でも、**ユーザー視点**を持つことが重要。「自分が良いと思う」ではなく「ユーザーが使いやすい」を目指す。

### 5. 非機能要件の理解

#### 機能だけでは不十分

**機能要件**: 「何ができるか」
- Aboutセクションがある
- Skillsセクションがある
- Projectsセクションがある

**非機能要件**: 「どう実現するか」
- アクセシビリティ（alt属性、コントラスト比）
- パフォーマンス（軽量な画像）
- デザイン（一貫性のある配色）

**実例**:
```markdown
### アクセシビリティ
- alt属性の適切な使用
- コントラスト比の確保
- 論理的なタブ順序
```

**学び**:
- 「動けばいい」ではなく、「誰でも使える」を目指す
- アクセシビリティは後付けではなく、最初から設計に組み込む
- 非機能要件は「見えない品質」を決める

### 6. スコープ管理の技術

#### MVPの考え方

**MVP (Minimum Viable Product)**: 最小限の機能で価値を提供する製品

**今回のMVP**:
- ✅ 静的なポートフォリオサイト
- ✅ HTML/CSSのみ
- ✅ シングルページ
- ✅ 基本的なホバーエフェクト

**将来の拡張**:
- レスポンシブデザイン（第2週）
- JavaScriptによるインタラクション（第3週以降）
- お問い合わせフォーム（バックエンド連携）

**学び**:
- **完璧を目指さない**（完璧主義の罠を避ける）
- 今週の学習目標に集中する
- 「今作るべきもの」と「後で追加できるもの」を区別する

### 7. 実践的な技術選択

#### なぜFlexboxなのか？

**技術選択の理由**:
1. **学習曲線**: Flexboxは理解しやすい（1次元レイアウト）
2. **実用性**: 実務で頻繁に使われる
3. **段階性**: CSS Gridより先に学ぶべき
4. **練習機会**: ナビ、スキル、カードなど複数箇所で使える

**Grid vs Flexbox**:
- **Flexbox**: 1次元レイアウト（行または列）
- **Grid**: 2次元レイアウト（行と列の両方）

**学習順序**:
1. Flexbox（今週）
2. Grid（後の週）

**学び**: 技術選択には段階がある。「新しい」より「基本的」を優先。

## まとめ: 要件定義書から学ぶべき本質
